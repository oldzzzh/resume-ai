 ## 最高准则
 **凡是能自己做的事，自己动手。** 不让用户操作，除非不可逾越的权限限制。用户的每一分钟应该花在产品决策和内容上，不是在命令行。
 
 ## 文件传输
 代码走 GitHub API push → 用户 `git pull`。一次 commit 包含完整改动，不分多次写文件。必须让用户操作时，一次给完整可执行的命令。
 
 ## 已踩过的坑（不要再犯）
 - mammoth 不能在浏览器 `import()`，必须走 API 路由转发
 - .git 目录被沙箱创建后外部无权写入，重新 git init 解决
 - Node REPL 变量名用 var 会在调用间冲突，每次用不同命名
 - Add-Content 分多次写文件会导致截断/乱码，全程一次写
 
 ## 架构原则
 - 包能不能在浏览器跑？不能就走 API 路由
 - 一次仅改一个功能点，验证通过再继续
 - API key 放 .env.local，不提交 Git
 - 端口固定 3000
 - 不引入不必要的依赖，优先用 Next.js 内置能力
 - 对于简历/JD相关文案优化，DeepSeek 模型指令需明确要求"和原文保持一致的语种"
 
 ## 沟通偏好
 - 技术术语第一次出现时用大白话解释（如"JD就是职位描述"）
 - 回答先给结论，再给理由，最后给细节
 - 不说漂亮话，不画饼，不回避问题
 - 回答控制在50-70行以内
 
 ## 内容创作支持
 - 选题要反映真实过程（踩坑、数据、决策过程）
 - 视频结构：前3秒钩子→过程→结果/数据
 - 封面字要大，信息要直接
 - 优先发小红书和B站，海外平台辅助
